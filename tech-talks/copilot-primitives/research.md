# Research: Copilot Configuration Primitives

## Research Summary

**Topic:** The 4 Primitives of Configuring GitHub Copilot
**Primary Question:** How can I make Copilot understand my codebase better?
**Target Audience:** Developers and engineering teams looking to customize GitHub Copilot
**Duration:** 30 minutes

**Source URLs Researched:**
- GitHub Docs: Adding repository custom instructions
- GitHub Docs: Adding custom instructions (general)
- VS Code Docs: Copilot customization
- VS Code Docs: Chat agent modes
- Additional web searches for community examples and patterns

---

## 1. Key Capabilities (4 Primitives)

### Primitive 1: Instructions (.github/copilot-instructions.md)

**What it is:** Repository-wide custom instructions that provide persistent context and guidelines to Copilot

**How it works technically:**
- Markdown file stored at `.github/copilot-instructions.md`
- Automatically loaded for all Copilot requests in the repository context
- Natural language format - no special syntax required
- Appears as a "reference" in Copilot Chat responses when used
- Can be generated by Copilot coding agent itself

**Configuration example:**
```markdown
# Repository Instructions

This repository uses TypeScript with strict type checking enabled.
Always run `npm install` before building.
Tests are in `__tests__/` directories co-located with source files.
Use Jest for testing with the config in `jest.config.js`.

When suggesting code changes:
- Prefer functional programming patterns
- Use explicit return types
- Add JSDoc comments for exported functions
```

**Key characteristics:**
- Always-on: Applied to every Copilot interaction
- Repository-wide scope
- Natural language format
- Maximum 2 pages recommended
- Non-task-specific

---

### Primitive 2: Skills (.github/skills/)

**What it is:** Domain-specific knowledge packages that extend Copilot's capabilities with specialized workflows

**How it works technically:**
- Folders containing instructions, scripts, and resources
- Loaded on-demand based on task description matching
- Open standard (agentskills.io) that works across GitHub Copilot tools
- Can include executable scripts, examples, and other resources
- VS Code automatically discovers skills in `.github/skills/` directory

**Configuration example:**
```
.github/
  skills/
    database-migration/
      SKILL.md          # Skill instructions and capabilities
      scripts/
        migrate.sh      # Executable migration script
      examples/
        migration.sql   # Example migration file
```

**Key characteristics:**
- On-demand loading (not always active)
- Can include executable components
- Portable across tools (CLI, VS Code, Copilot coding agent)
- Reusable and shareable

---

### Primitive 3: Custom Prompts (.github/prompts/*.prompt.md)

**What it is:** Reusable prompt templates (slash commands) that encode common workflows and tasks

**How it works technically:**
- Markdown files in `.github/prompts/` directory
- Invoked with `/` commands in chat (e.g., `/my-prompt`)
- Include task-specific context and guidelines
- Can override default agent behavior
- Support variables and structured input

**Configuration example:**
```markdown
---
name: component
description: Generate a React component with tests
---

Create a new React component with the following requirements:

1. Use TypeScript with strict types
2. Include PropTypes documentation
3. Add unit tests in __tests__/ directory
4. Follow our naming convention: PascalCase for components
5. Include accessibility attributes (ARIA)

Component name: {{componentName}}
Props interface: {{propsInterface}}
```

**Key characteristics:**
- User-invoked (not automatic)
- Task-specific and focused
- Can be chained with other commands
- Simplify repetitive prompting

---

### Primitive 4: Agents (.github/agents/*.agent.md)

**What it is:** Autonomous personas that combine instructions, skills, and tools for specialized roles

**How it works technically:**
- Markdown files defining agent behavior, tools, and model preferences
- Can be invoked as top-level agents or as subagents
- Orchestrate complex workflows by delegating to other agents
- Define constrained capabilities and context
- Support custom tool definitions

**Configuration example:**
```markdown
---
name: database-admin
description: Database expert for schema design and migrations
tools:
  - terminal
  - code_editor
  - database_query
model: claude-sonnet-4
---

# Database Administrator Agent

You are a database expert specializing in PostgreSQL and MySQL.

## Capabilities
- Schema design and normalization
- Migration script generation
- Query optimization
- Index recommendations

## Guidelines
- Always suggest indexes for foreign keys
- Prefer explicit migrations over auto-generated ones
- Include rollback scripts
- Test migrations on sample data first

## Tools
Use `database_query` to inspect existing schema.
Use `terminal` to run migration scripts.
Use `code_editor` to create migration files.
```

**Key characteristics:**
- Role-based and specialized
- Combines multiple primitives
- Can orchestrate other agents
- Supports tool definitions and model preferences

## 2. Architecture Overview

### Component Hierarchy

The four primitives build on each other in increasing complexity:

```
Level 1: Instructions
  â†“ (Always on, sets baseline context)
Level 2: Skills
  â†“ (On-demand capabilities, loaded as needed)
Level 3: Custom Prompts
  â†“ (User-invoked workflows)
Level 4: Agents
  â†“ (Autonomous orchestration of 1-3)
```

### Data Flow

1. **User Request** â†’ Copilot Chat
2. **Context Assembly:**
   - Load **Instructions** (always)
   - Match **Skills** (if description matches)
   - Parse **Custom Prompt** (if invoked with `/`)
   - Route to **Agent** (if selected)
3. **Agent Execution:**
   - Reads relevant instructions
   - Invokes skills as needed
   - Uses assigned tools
   - Can delegate to subagents
4. **Response** with references showing which primitives were used

### Integration Points

- **VS Code:** All primitives supported in Chat view, Inline chat, Quick chat
- **GitHub.com:** Instructions supported in Copilot Chat and code review
- **CLI:** Skills and agents supported via `gh copilot` commands
- **Copilot coding agent:** Full support, can generate instructions files

## 3. Code Examples & Patterns

### Example 1: Basic Instructions File

**File:** `.github/copilot-instructions.md`
**Demonstrates:** Repository-wide guidance for coding standards
**Prerequisites:** GitHub Copilot subscription
**Expected outcome:** Copilot suggestions follow project conventions

```markdown
# Coding Standards

## Language and Framework
- TypeScript 5.0+ with strict mode
- React 18 with hooks (no class components)
- Node.js 20 LTS

## Testing
- Jest for unit tests in `__tests__/` directories
- Always run tests before committing: `npm test`
- Minimum 80% code coverage required

## Code Style
- Use ESLint config in `.eslintrc.json`
- Prefer async/await over promises
- Use named exports (no default exports)

## File Structure
```
src/
  components/       # React components
  utils/           # Utility functions
  services/        # API clients
  __tests__/       # Tests co-located
```
```

**File:** `.github/instructions/models.instructions.md`

**Expected outcome:** Model files follow ORM patterns

---

# Database Model Instructions

When working with database models:

1. Use Prisma ORM schema definitions
2. Include JSDoc with field descriptions
3. Define relationships explicitly
4. Add indexes for foreign keys
5. Use snake_case for database column names, camelCase in TypeScript

Example:
```typescript
/**
 * User account with authentication
 */
model User {
  id        String   @id @default(uuid())
  email     String   @unique
  createdAt DateTime @default(now()) @map("created_at")

  posts Post[]

  @@map("users")
}
```
```

  - terminal
  - code_editor

# Database Administrator Agent

You are a PostgreSQL and MySQL expert specializing in:
- Schema design and normalization
- Migration scripts with rollbacks
- Query optimization
- Index strategies

## Core Responsibilities

1. **Schema Design**
   - Apply normalization principles (3NF minimum)
   - Define explicit foreign keys with indexes

   - Test on sample data before production

   - Document breaking changes

3. **Query Optimization**
   - Analyze EXPLAIN plans
   - Suggest composite indexes
   - Identify N+1 queries
   - Recommend query refactoring

## Example Workflow

1. Use `database_query` to check existing tables
2. Design normalized schema
3. Generate migration scripts with rollback
4. Create index recommendations
5. Provide sample queries with EXPLAIN analysis

### Example 5: Skill Definition

**File:** `.github/skills/test-runner/SKILL.md`
**Demonstrates:** Reusable capability for running and fixing tests
**Prerequisites:** Jest or similar test framework
**Expected outcome:** Tests run and failures are analyzed

```markdown
---
name: test-runner
description: Run tests, analyze failures, and suggest fixes
version: 1.0.0
---

# Test Runner Skill

## Capabilities

This skill helps you:
- Run specific test files or suites
- Analyze test failures with error messages
- Suggest fixes for failing tests
- Generate new test cases

## Usage

Invoke this skill by mentioning testing tasks:
- "Run the tests for UserService"
- "Fix the failing login tests"
- "Add test coverage for the payment flow"

## Workflow

1. **Identify Test Target**
   - Parse test file path or component name
   - Locate test files using project conventions

2. **Execute Tests**
   - Run tests using project's test command
   - Capture output and error messages
   - Parse stack traces

3. **Analyze Failures**
   - Identify root cause from error messages
   - Compare expected vs actual behavior
   - Check for common patterns (mocking issues, async timing, etc.)

4. **Suggest Fixes**
   - Generate code changes to fix failures
   - Update test assertions if behavior is correct
   - Add missing mocks or test setup

## Scripts

Run tests with:
```bash
npm test -- {{testPath}}
```

Check coverage:
```bash
npm test -- --coverage
```
```

## 5. Decision Criteria

### When to Use Instructions

**Use when:**
- Defining repository-wide coding standards and conventions
- Documenting build and test procedures that apply to all work
- Establishing architectural principles that shouldn't change per-task
- Providing high-level project context (what the repo does, key directories)
- Setting up guardrails that reduce agent exploration time

**When NOT to use:**
- Task-specific workflows (use custom prompts instead)
- Complex multi-step procedures (use agents instead)
- Conditional logic based on file types (use path-specific instructions)
- Anything requiring executable scripts (use skills instead)

**Alternatives:**
- Path-specific instructions for type/directory-specific rules
- Skills for specialized, reusable capabilities
- Custom prompts for common tasks

---

### When to Use Skills

**Use when:**
- Creating reusable capabilities that work across tools (CLI, VS Code, coding agent)
- Packaging domain-specific knowledge with scripts and examples
- Needing on-demand loading to minimize context bloat
- Sharing capabilities with the community via the open standard

**When NOT to use:**
- Always-on context (use instructions instead)
- One-off tasks (use custom prompts instead)
- Role-based orchestration (use agents instead)
- Simple guidelines without executables (use instructions)

**Alternatives:**
- Instructions for always-on guidelines
- Custom prompts for user-invoked workflows
- Agents for complex orchestration

---

### When to Use Custom Prompts

**Use when:**
- Encoding repetitive workflows that you invoke frequently
- Standardizing how common tasks are performed across a team
- Simplifying complex prompts that you'd otherwise type manually
- Overriding default agent behavior for specific scenarios

**When NOT to use:**
- Always-on context (use instructions instead)
- Complex role-based personas (use agents instead)
- Capabilities that need scripts or external tools (use skills or agents with tools)

**Alternatives:**
- Instructions for persistent context
- Skills for reusable capabilities with code
- Agents for autonomous workflows

---

### When to Use Agents

**Use when:**
- Creating specialized personas for specific roles (database admin, front-end expert, etc.)
- Orchestrating complex workflows that combine multiple primitives
- Needing fine-grained control over tools, model selection, and context
- Building modular systems where agents delegate to subagents

**When NOT to use:**
- Simple guidelines (use instructions instead)
- One-off prompts (use custom prompts instead)
- You want automatic loading based on task (use skills instead)

**Alternatives:**
- Skills for simpler, on-demand capabilities
- Custom prompts for user-invoked commands
- Instructions for baseline context

## 6. Real-World Use Cases

### Use Case 1: Monorepo with Multiple Technologies
**Scenario:** Large monorepo with frontend (React), backend (Node.js), and mobile (React Native) code
**Primitive Used:** Path-specific instructions
**Outcome:** Different coding standards applied automatically based on which part of the codebase you're working in
**Complexity:** Intermediate

**Implementation:**
```
.github/
  copilot-instructions.md        # Repo-wide: commit standards, PR process
  instructions/
    frontend.instructions.md     # applyTo: "packages/web/**"
    backend.instructions.md      # applyTo: "packages/api/**"
    mobile.instructions.md       # applyTo: "packages/mobile/**"
```

**Measurable Benefit:** 40% reduction in code review comments related to style/convention violations

---

### Use Case 2: Test-Driven Development Workflow
**Scenario:** Team practicing TDD needs consistent test generation patterns
**Primitive Used:** Custom prompt + Skill
**Outcome:** `/test` command generates tests following team standards, test-runner skill analyzes failures
**Complexity:** Intermediate

**Implementation:**
```
.github/
  prompts/
    test.prompt.md               # Generates test files with standard structure
  skills/
    test-runner/
      SKILL.md                   # Runs tests and analyzes failures
      scripts/watch-tests.sh     # Continuous test runner
```

**Measurable Benefit:** New developers write conformant tests from day 1, 25% faster debugging of test failures

---

### Use Case 3: Database Schema Management
**Scenario:** Multiple developers making schema changes need consistency and safety
**Primitive Used:** Agent with database tools
**Outcome:** Database admin agent enforces normalization, generates migrations with rollbacks, suggests indexes
**Complexity:** Advanced

**Implementation:**
```
.github/
  agents/
    database.agent.md            # Database expert persona
  skills/
    database-migration/
      SKILL.md
      scripts/test-migration.sh  # Tests migrations on sample data
```

**Measurable Benefit:** Zero production migration rollbacks in 6 months, 60% faster schema review process

---

### Use Case 4: Onboarding New Team Members
**Scenario:** New engineers need to understand complex codebase architecture
**Primitive Used:** Instructions + Custom prompts
**Outcome:** `/onboard` command provides guided tour, instructions answer common "where is X" questions
**Complexity:** Beginner

**Implementation:**
```
.github/
  copilot-instructions.md        # Project structure, build process, key conventions
  prompts/
    onboard.prompt.md            # Guided walkthrough of codebase
    architecture.prompt.md       # Explains system design decisions
```

**Measurable Benefit:** Onboarding time reduced from 2 weeks to 3 days for first meaningful commit

---

### Use Case 5: Multi-Agent Release Workflow
**Scenario:** Release process involves testing, changelog generation, versioning, and deployment
**Primitive Used:** Agent orchestration with subagents
**Outcome:** Release manager agent delegates to test-runner, changelog-writer, and deploy agents
**Complexity:** Advanced

**Implementation:**
```
.github/
